# Changelog

All notable changes to BBoeOS are documented in this file. Dates reflect
when changes landed, grouped under the version that was (or will be) current
at the time.

## [Unreleased](https://github.com/bboe/BBoeOS/compare/5156ae9...main)

- asm.c: port ``symbol_set`` / ``symbol_add_constant`` to pure C.  ``symbol_set`` becomes a regparm(1) function taking value (AX) + scope (stack): runs ``symbol_lookup(scope)`` first, then either writes the value word via ``far_write16`` at the existing entry (hit) or calls pure-C ``symbol_add(value, scope)`` (miss) and caches ``symbol_count - 1`` as the new entry's index.  ``symbol_add_constant`` becomes a regparm(1) function: calls ``symbol_set(value, 0xFFFF)`` then flips the type byte at ``entry + SYMBOL_NAME_LENGTH + 2`` to 1 via ``far_write8``.  The ``symbol_set_global`` / ``symbol_set_local`` always_inline wrappers update their naked-asm bodies to match the new ``push word <scope>; call symbol_set; add sp, 2`` stack-arg ABI.  The ``symbol_add_constant_c`` regparm wrapper retires — both its call sites call ``symbol_add_constant`` directly.  Two obsolete globals (``_g_symbol_set_value`` / ``_g_symbol_set_scope``) retire as well, since symbol_set's pure-C body spills its args to the bp frame instead of stashing them through named globals.  The else-branch's offset expression is written inline (``far_write16(symbol_entry_address(...) + SYMBOL_NAME_LENGTH, value)``) rather than through a local ``offset`` — a named local triggered cc.py's pin-tracking bug where the auto-pinned local stayed tracked as "held in AX" even after a subsequent ``mov ax, [bp-2]`` overwrote AX with a different value.  Binary grows 74 bytes (11674 → 11748).  Self-host test still byte-identical on all 31 static programs

- cc.py: add ``far_read8`` / ``far_read16`` / ``far_write8`` / ``far_write16`` builtins; port ``symbol_lookup`` to pure C.  The builtins emit ``[es:<offset>]``-prefixed memory accesses (``mov al, [es:bx]``, ``mov word [es:bx], <value>``, etc.) so C code can reach data that lives in the symbol segment rather than DS.  Each takes a single offset expression (regparm(1)-style into BX); the write variants also take a value (constant fast path or push/pop AX around offset eval).  Targeted for protected-mode retargeting: swapping the body of each builtin to a plain ``[<offset>]`` load/store (no segment prefix) leaves the calling C untouched.  asm.c's ``symbol_lookup`` body (41 lines of inline asm doing a linear scan over ES:0..EFF8) becomes a pure-C ``regparm(1)`` function that uses the new builtins for the entry-byte reads.  Return ABI flips from dual AX+CF (value on hit, CF signaling miss) to AX-only + ``last_symbol_index`` sentinel (0xFFFF on miss) — all two remaining asm callers (``symbol_set`` body + ``symbol_lookup_c`` wrapper) already test ``last_symbol_index`` rather than CF.  The inner name-compare loop walks source_cursor (SI pin) via ``source_cursor[0]`` + ``source_cursor = source_cursor + 1``; reading the subscript with a variable offset would make cc.py emit ``add si, <reg>`` and wreck the SI alias, so the port saves ``source_cursor`` at entry and restores it after each inner compare.  ``symbol_lookup_c`` wrapper retires — six call sites now call ``symbol_lookup`` directly.  Binary grows 62 bytes (11612 → 11674): bp frame + 5 locals + ``far_read8`` call overhead per byte compare outweigh the compact asm's ``cmp al, [es:di]``.  Self-host test still byte-identical on all 31 static programs

- cc.py: iterate ``apply_defines`` to fixed point; asm.c main pointer-init goes pure C.  The previous one-pass substitution didn't recursively expand transitive ``#define``s: ``#define OUTPUT_BUFFER (LINE_BUFFER + 256)`` followed by ``#define LINE_BUFFER _program_end`` left the inner ``LINE_BUFFER`` unresolved.  A fixed-point loop (bounded at 16 rounds to guard against self-reference cycles) now walks the token stream until no substitution happens.  ``_program_end`` joins cc.py's NAMED_CONSTANTS and ``tests/bboeos.h`` gains ``extern char _program_end[];`` for clang's syntax check.  asm.c's ``main()`` replaces its 4-instruction ``asm("mov word [_g_line_buffer], LINE_BUFFER\n...")`` block with the pure-C form ``line_buffer = LINE_BUFFER; output_buffer = OUTPUT_BUFFER; source_buffer = SOURCE_BUFFER; include_source_save = SOURCE_BUFFER + 512;`` — one fewer inline-asm block in the file.  Binary unchanged (11612 bytes; the generated mov/word stores match the retired asm byte-for-byte).  Self-host test still byte-identical on all 31 static programs

- asm.c: mark 9 more naked-asm helpers ``always_inline``.  ``close_source``, ``open_file_ro``, ``symbol_add_constant_c``, ``symbol_set_global``, ``symbol_set_local``, ``symbol_lookup_c``, ``mnemonic_keyword_at``, ``mnemonic_dispatch_at``, and ``read_source_sector`` all get the attribute; each had 1-3 call sites and a small enough body that inlining wins after dropping the free-standing function body.  ``open_file_ro``'s internal ``.ofr_ok`` label gets per-site uniquification (``.ofr_ok_inl2``, ``.ofr_ok_inl3``) so its two inlinings don't clash.  Binary shrinks 85 bytes (11697 → 11612).  Self-host test still byte-identical on all 31 static programs

- cc.py: add ``__attribute__((always_inline))`` for naked-asm helpers.  The attribute is valid on function definitions whose body is a single ``asm("...")`` statement and whose parameter list is empty or one regparm(1) arg (stack args would need a caller-side ``add sp, N`` cleanup that doesn't fit inside an inlined body).  At each C-level call site, cc.py splices the body text with per-site local-label uniquification (``.foo`` → ``.foo_inl<N>``) so multiple inlinings don't clash; C escape sequences in the body are decoded via the same path ``builtin_asm`` uses, so ``asm("push ds\npop es")`` lays down two lines.  No free-standing function body is emitted — inline-asm ``call <name>`` would fail to resolve, which is the expected signal to the author.  Applied to ``restore_es`` in asm.c: binary shrinks 10 bytes (11707 → 11697) from retiring 5 call sites' 3-byte ``call`` + 1-byte shared ``ret`` + 2-byte body in favor of 2-byte inlined ``push ds / pop es`` at each site.  Self-host test still byte-identical on all 31 static programs

- asm.c: add ``close_source`` / ``open_file_ro`` helpers, port do_pass / include_push / include_pop syscall blocks to pure C.  ``close_source`` is a 3-line naked-asm thunk that issues SYS_IO_CLOSE on ``source_fd`` through the ES-safe ``syscall`` wrapper; ``open_file_ro(char *path)`` takes the path via regparm(1) AX, issues SYS_IO_OPEN, and returns the fd or -1 on error.  ``do_pass`` replaces its open+branch and close asm blocks with ``source_fd = open_file_ro(source_name); if (source_fd == -1) { error_flag = 1; return; } ... close_source();``; ``include_push`` replaces its open+branch asm with the same pattern on ``include_path``; ``include_pop`` replaces its close asm with a direct ``close_source();`` call.  Writing the fd straight to ``source_fd`` (rather than through an ``int fd`` local) avoids the bp frame cc.py would otherwise add.  Binary grows 24 bytes (11683 → 11707): two helper bodies + call-site overhead slightly outweigh the retired inline-asm.  Four asm blocks in hot-path bodies move into two small helpers — net -2 inline-asm blocks in the file.  Self-host test still byte-identical on all 31 static programs

- asm.c: consolidate four ``asm("push ds\npop es");`` ES-restore blocks into a shared ``restore_es()`` helper.  Each site (``abort_unknown_impl``, ``die_error_pass1_io``, ``die_error_pass1_iter``, ``die_symbol_overflow``, plus ``main``'s pre-close ritual) had the same 2-byte sequence to point ES back at DS before the kernel-expecting-ES=0 builtins (``printf``, ``die``, ``close``) run.  Collapsing to a single naked-asm helper called by all five sites removes four inline-asm blocks at a 1-byte binary cost (the helper's body + ret vs. the inline ``push ds ; pop es`` per call site — roughly break-even; 11682 → 11683).  Self-host test still byte-identical on all 31 static programs

- asm.c: replace include_push's SI-stash asm with a pure-C assignment.  ``asm("mov [_g_include_push_arg], si");`` existed because SI (= ``source_cursor``) carried the raw include filename pointer on entry from parse_directive; cc.py's codegen would have clobbered SI before the C body could read it.  Since ``source_cursor`` is the SI-pinned global, ``include_push_arg = source_cursor;`` compiles to the identical store and reads the same value.  Binary shrinks 1 byte (11683 → 11682).  Self-host test still byte-identical on all 31 static programs

- asm.c: add ``symbol_set_global`` / ``symbol_set_local`` wrappers; replace 4 ``symbol_set`` inline-asm blocks with pure-C calls.  Both wrappers are regparm(1) naked-asm thunks parallel to the existing ``symbol_add_constant_c`` / ``symbol_lookup_c``: ``value`` arrives in AX, the wrapper hardcodes BX (0xFFFF for ``_global``, ``[_g_global_scope]`` for ``_local``), then calls ``symbol_set``.  ``handle_unknown_word`` and ``parse_line`` each had two identical ``mov ax, [_g_current_address] ; mov bx, scope ; call symbol_set`` sequences (one per is_local branch); those collapse to ``symbol_set_local(current_address);`` / ``symbol_set_global(current_address); global_scope = last_symbol_index;``.  Binary grows 25 bytes (11658 → 11683): ~24 bytes for the two new wrapper bodies plus cc.py's pin-save shims at the call sites slightly outweigh the compact AX/BX setup the retired inline-asm had.  Self-host test still byte-identical on all 31 static programs

- asm.c: replace 4 ``symbol_lookup`` inline-asm blocks with ``symbol_lookup_c(scope)`` calls.  ``handle_unknown_word`` and ``parse_line`` each had a ``mov bx, 0xFFFF ; call symbol_lookup ; jc .no_update ; mov [_g_global_scope], ax`` sequence that collapses to ``symbol_lookup_c(0xFFFF); if (last_symbol_index != 0xFFFF) global_scope = last_symbol_index;`` (the CF-branch becomes a memory test of ``last_symbol_index`` which ``symbol_lookup`` sets on every path).  ``peek_label_target``'s two branches become ``peek_label_value = symbol_lookup_c(scope);`` after a defensive ``last_symbol_index = 0xFFFF`` pre-set that the retired asm did inline.  Binary grows 3 bytes (11655 → 11658): bp-frame/pin-save shims that cc.py adds around the C calls slightly outweigh the tighter hand-written asm.  Self-host test still byte-identical on all 31 static programs

- asm.c: replace 4 ``asm("call X");`` blocks with direct C calls.  ``do_pass`` calls ``parse_line()`` directly, ``run_pass1`` / ``run_pass2`` call ``do_pass()`` directly, and ``main`` calls ``flush_output()`` directly.  All four callees are pure C; cc.py resolves the call through its user_functions registry, so the generated ``call X`` sequence is byte-identical to the explicit asm form.  Binary unchanged (11655 bytes).  Self-host test still byte-identical on all 31 static programs

- asm.c: retire the 31 archaeological ``<name> equ _g_<name>`` aliases in the tail asm block.  With every handler now pure C and the remaining inline-asm blocks all using the explicit ``_g_<name>`` form directly, no source-level reference relied on the bare-name aliases.  The alias block + surrounding explanatory comments retire; binary size unchanged (aliases are NASM-time constructs).  Verified by grep: zero occurrences of each bare name inside double-quoted ``asm(...)`` content — every hit was either the alias definition itself or prose.  Self-host test still byte-identical on all 31 static programs

- asm.c: remove dead ``make_modrm_reg_reg`` thunk.  The "legacy label" whose ~7 inline-asm callers pushed BX as the stack-passed ``rm`` has zero remaining callers after every handler migrated to calling ``make_modrm_reg_reg_impl`` from pure C; the 4-instruction thunk body + surrounding docstring retire.  Binary shrinks 8 bytes (11663 → 11655)

- asm.c: port ``flush_output`` to pure C.  Early-returns on ``output_position == 0`` in C, then issues ``SYS_IO_WRITE`` through the ES-safe ``syscall`` wrapper in a small inline-asm block, and resets ``output_position = 0`` in C.  Drops the retired asm's ``push ax ; push cx ; push si ; push di`` / ``pop`` chain: emit_byte_al already does ``pusha`` / ``popa`` externally around the entire emit_byte chain (including the transitive flush_output call when the buffer fills), so transitive callers are still protected; direct pure-C callers of ``emit_byte`` don't rely on callee register preservation (cc.py's calling convention spills everything it cares about).  Binary shrinks 3 bytes (11666 → 11663): the 8-byte save/restore chain retires in favor of a bp frame plus a single ``cmp word [_g_output_position], 0 / je`` test.  Self-host test still byte-identical on all 31 static programs

- asm.c: port ``load_src_sector`` to pure C.  A naked-asm ``read_source_sector`` helper wraps the ``SYS_IO_READ`` call (``BX = source_fd``, ``DI = SOURCE_BUFFER``, ``CX = 512``) through the ES-safe ``syscall`` wrapper and returns AX = bytes read (or -1 on error); factoring it out lets the caller receive the count through the standard return-in-AX convention since cc.py has no syntax for binding an inline ``call syscall``'s AX to a C local.  ``load_src_sector`` itself becomes a plain C body: ``int bytes = read_source_sector(); if (bytes == 0) return 1; if (bytes == -1) return 1; source_buffer_valid = bytes; source_buffer_position = 0; return 0;``.  Binary grows 24 bytes (11642 → 11666): the bp frame + ``bytes`` spill + two ``== 0`` / ``== -1`` compares vs. the retired asm's 3-branch shared-tail form.  Self-host test still byte-identical on all 31 static programs

- asm.c: retire ``parse_register_c`` / ``parse_register_optional`` forwarders.  Both collapsed to identical ``return parse_register();`` one-liners in the parse_register port; with no remaining ABI difference between the two (or between either and ``parse_register`` itself), the wrappers were just cosmetic.  All 16 ``parse_register_c()`` + 7 ``parse_register_optional()`` call sites now call ``parse_register()`` directly.  Binary shrinks 16 bytes (11658 → 11642): two bp-frame epilogues + two naked-call shims retired vs. the modest call-site overhead (pure bytes, no runtime change).  Self-host test still byte-identical on all 31 static programs

- asm.c: port ``match_word`` to pure C and retire the four wrapper shims.  The pure-C body takes the keyword via ``regparm(1)`` AX (same ABI the old ``match_word_c`` wrapper exposed to C callers) and signals match/miss via ``carry_return``; source cursor is advanced on match / restored on miss via a ``char *saved`` local.  Keyword strings in the assembler's tables are all lowercase, so only the source side gets case-folded — one fewer branch than the retired asm's dual-lowercase idiom.  The four wrapper shims (``match_word_c(char *keyword)`` generic + three ``match_word_c_str_equ`` / ``_short`` / ``_word`` specialized) all retire; 16 call sites rename ``match_word_c(X)`` / ``match_word_c_str_X()`` → ``match_word(X)``.  With ``parse_mnemonic`` already in pure C, ``match_word`` no longer has any asm-side caller.  Binary grows 26 bytes (11632 → 11658): the pure-C body's bp frame + ``saved`` / ``s`` locals + cc.py's SI-scratch guard around each ``keyword[0]`` / ``source_cursor[0]`` read pay a net tax against the savings from retiring the four wrapper call sequences (each was a ``push ax ; mov ax, STR_X ; call match_word_c ; pop ax`` shape, now collapsed to a single ``mov ax, STR_X ; call match_word`` at the C call site).  5 inline-asm blocks retired

- asm.c: port ``parse_mnemonic`` to pure C.  Two tiny ``regparm(1)`` naked-asm helpers (``mnemonic_keyword_at`` / ``mnemonic_dispatch_at``) factor out the two idioms cc.py can't express in C — reading a 16-bit keyword pointer out of the packed ``dw STR_X, handle_X`` data table, and ``call [bx+2]`` through the handler pointer — so the dispatcher itself becomes a clean ``while (1)`` loop: fetch the keyword at the current index, NULL → ``handle_unknown_word()``, else ``match_word_c(keyword)`` → ``mnemonic_dispatch_at(index)``, else increment index.  The two helpers each compute ``mnemonic_table + index * 4`` redundantly rather than sharing a single cursor — simpler than threading a global entry pointer, and cc.py's regparm(1) keeps the shift+add chain in-register.  Binary grows 66 bytes (11566 → 11632): the pure-C loop's bp frame + ``index`` spill + the helpers' bp-frame-free naked-asm bodies pay for themselves against the retired asm's four-register save/restore pair.  1 inline-asm block retired in the caller (two new 4-line helpers added)

- asm.c: port ``parse_register`` to pure C.  The 60-line asm body's linear scan over ``register_table`` (4-byte packed entries: 2-char name + reg-num + size, zero-terminated) becomes a ``while (entry[0] != '\0')`` loop with a case-insensitive 2-char compare and an identifier-boundary check on ``source_cursor[2]``.  Return ABI flips from CF + packed AX to a sentinel: ``(size << 8) | reg`` on match with ``source_cursor`` advanced past the 2-char name, ``-1`` on miss with ``source_cursor`` unchanged.  ``parse_register_c`` (previously a ``carry_return`` naked wrapper) and ``parse_register_optional`` (previously a ``jnc`` → ``-1`` converter) collapse into identical ``return parse_register();`` one-liners — kept as two separate names so call sites still read ``_c`` for "assume success" vs ``_optional`` for "check ``pr >= 0``".  cc.py: ``register_table`` joins ``NAMED_CONSTANTS`` so the pure-C body can reference the NASM label directly as a ``char *``; ``tests/bboeos.h`` gains the matching ``extern char register_table[];`` for clang's syntax check.  Binary grows 102 bytes (11464 → 11566): bp frame + cc.py's SI-scratch guard (``push si ; mov si, cx ; mov al, [si] ; pop si``) around each ``entry[N]`` read because ``entry`` auto-pins to CX and CX isn't a valid 16-bit base register, plus the char-class boundary ``||`` chain.  1 inline-asm block retired.  Self-host test still byte-identical on all 31 static programs

- cc.py: root-cause fix for ``_dedup_register_reloads`` dropping a needed ``mov si, ax`` store.  The peephole tracked the last value written to SI (as a string like ``"ax"``) and skipped subsequent ``mov si, ax`` lines when the source matched — but it didn't invalidate the tracked source when the source REGISTER itself got modified.  So ``mov si, ax / inc ax / mov si, ax`` collapsed to ``mov si, ax / inc ax``: the second store — which would have captured the incremented AX — got elided, leaving SI stale.  This hit cc.py-generated programs' ``mov [_g_history+si], ax`` emit in the self-hosted assembler's parse_operand port: the ``[disp+reg]`` detection's ``source_cursor = close; if (source_cursor[0] == ']') source_cursor = source_cursor + 1;`` compiled to the bad pattern and handle_mov's op2 parse saw a stale source cursor, returning type=1 (imm) for what was actually ``ax``.  Fix: when the tracked value is a non-memory source (register or immediate), scan each instruction for a target that matches the source register — ``add <reg>``, ``sub``, ``and``, ``or``, ``xor``, ``shl``, ``shr``, ``sar``, ``rol``, ``ror``, ``rcl``, ``rcr``, ``inc``, ``dec``, ``neg``, ``not``, ``mul``, ``imul``, ``div``, ``idiv``, ``mov`` — and invalidate the tracking.  Byte-half targets (``ax`` invalidated by ``al``/``ah`` writes) are caught via a ``target[0] == value[0]`` check.  ``src/c/asm.c`` parse_operand's ``[disp+reg]`` branch reverts to the natural ``source_cursor = close; if (...) source_cursor = source_cursor + 1;`` form instead of the ``next = close + 1; source_cursor = next;`` workaround, shaving 17 bytes (11481 → 11464)

- asm.c: port ``parse_operand`` to pure C.  The 186-line asm body's eight operand shapes — ``byte`` / ``word`` size prefixes, bare register, immediate (via ``resolve_value``), ``[disp16]``, ``[reg]`` / ``[reg+disp]``, ``[disp+reg]`` (NASM dialect detected by scanning backwards from ``]`` for a trailing register preceded by ``+``), ``[reg-disp]``, and ``[es:...]`` segment overrides (emitting ``0x26`` before returning so the next emit continues the instruction) — collapse into a linear structured dispatch.  Return ABI: ``(type << 8) | reg`` in AX with ``parse_operand_value`` carrying the disp/imm as a side effect global; ``op1_size`` stays a side-effect global for the register and size-prefix paths.  New ``STR_BYTE`` joins cc.py's NAMED_CONSTANTS so ``match_word_c(STR_BYTE)`` handles the size prefix probe without another specialized wrapper.  ``parse_operand_c`` becomes a thin ``return parse_operand();`` identity wrapper — callers keep their existing ``int po = parse_operand_c();`` pattern while the DX-stash that the retired inline-asm version did is now redundant (pure-C parse_operand writes ``parse_operand_value`` directly).  The ``[disp+reg]`` branch also had to rewrite ``source_cursor = close; if (source_cursor[0] == ']') source_cursor = source_cursor + 1;`` into the equivalent ``char *next = close; if (close[0] == ']') next = close + 1; source_cursor = next;`` form because cc.py's ``source_cursor = X`` codegen drops the write-back ``mov si, ax`` when the next statement is a conditional ``source_cursor++`` inside a C ``if`` body — a real codegen bug that only surfaced on the ``[SYM+si], AX`` pattern cc.py-generated programs use heavily, caught by the gdemo test's ``mov [_g_history+si], ax`` emit dropping to ``mov word [_g_history-2+si], 0`` (C7 path) instead of ``mov [_g_history+si], ax`` (89 path).  Binary grows 157 bytes (11324 → 11481): bp frame + 5 locals for the bracket-scan cursors + the ``end - bracket_start >= 2`` pointer-arithmetic subtract + the nested if-chains over the 8 operand shapes.  1 inline-asm block retired (186 → 0 lines)

- cc.py: two SI-scratch correctness fixes plus a loop-exit ``ax_local`` invalidation.  **_emit_byte_index_si** (the byte-compare fast path) now wraps the ``mov si, <base>`` with a push/pop guard when an ``asm_register("si")`` global exists — without it, code like ``while (close[0] != ']')`` lowered to ``mov si, [close local] ; cmp byte [si], 93`` clobbered the pinned ``source_cursor`` alias, so subsequent ``char *saved = source_cursor;`` captured whatever ``close`` happened to land at.  The helper now returns ``(operand, guarded)`` and every caller in ``_try_fuse_word_conditions`` / ``emit_comparison`` pairs the guard with a matching ``_si_scratch_guard_end`` after the ``cmp`` instruction.  **generate_while** now calls ``ax_clear()`` after the loop end-label so downstream code reloads from memory — a ``break`` inside ``if (prev[0] != ' ') break;`` leaves AX holding ``prev`` (= end-1), not the ``end`` tracked by ``ax_local``, so ``end - bracket_start`` was computing one-off before this fix.  No user-visible codegen change for programs without ``asm_register("si")`` globals; ``src/c/asm.c`` grows 6 bytes (11318 → 11324) from the added push/pop guards plus the forced reloads at loop exits.  These fixes unblock the in-flight ``parse_operand`` port which hits both bugs in its bracket-scanning loops and post-loop ``char *saved = source_cursor;`` stashes

- asm.c: port ``resolve_value`` to pure C.  The recursive expression evaluator handles ``(expr)`` paren grouping, ``'c'`` char literals, ``` `c` ``` backtick char literals (with ``\n`` / ``\0`` / ``\t`` / ``\r`` escape plus pass-through unknown-escape), ``$`` (current address), decimal / hex / binary numbers via ``parse_number``, and named symbols via a new ``symbol_lookup_c`` wrapper — followed by a right-associative ``+`` / ``-`` / ``*`` / ``/`` / ``&`` / ``|`` / ``^`` chain (flat precedence, matching the retired asm's shared ``.rv_expr_done`` fall-through).  New ``regparm(1) int symbol_lookup_c(int scope)`` naked-asm wrapper threads scope AX → BX then calls ``symbol_lookup`` (caller pre-loads SI = name via ``source_cursor = name_start;``); AX = value propagates through the naked ret.  After the symbol_lookup call the port restores ``source_cursor = end_pos`` so the operator chain sees the post-identifier position — the retired asm left SI at the null-terminated delim position via a pop-chain that my cc.py lowering couldn't reproduce naturally, and *not* advancing SI would leave the operator check reading the name's first letter and returning 0 for every ``SYMBOL + N`` expression (first debug run failed all 31 tests with 0-byte offsets everywhere).  Char-class check in the symbol-scan loop uses the retired asm's ``c >= 'a'`` no-upper-bound form so the char-class boundary stays byte-identical even for the unusual non-ASCII bytes that don't appear in real source.  Binary grows 190 bytes (11128 → 11318): bp frame + 5 locals (value, first, end_pos, delim, scope) + the five-branch primary dispatch + the seven-branch operator chain + recursion stack pressure + the new symbol_lookup_c wrapper (bp frame + 2-instruction body, ~10 bytes).  1 inline-asm block retired (219 → 0 lines of asm in the body)

- asm.c: port ``parse_directive`` to pure C.  The directive dispatcher's six keyword paths (``%assign`` / ``%define`` sharing a ``NAME = VALUE`` binding, ``%include``, ``org``, ``times N db ...``, ``db``, ``dw``, ``dd``, fall-through to ``parse_mnemonic``) collapse into a linear if-chain over new generic ``match_word_c(char *keyword)`` calls.  cc.py: eleven ``STR_*`` assembler-keyword labels (``STR_ASSIGN`` / ``STR_DB`` / ``STR_DD`` / ``STR_DEFINE`` / ``STR_DW`` / ``STR_EQU`` / ``STR_INCLUDE`` / ``STR_ORG`` / ``STR_SHORT`` / ``STR_TIMES`` / ``STR_WORD``) join the ``NAMED_CONSTANTS`` set so ``match_word_c(STR_ORG)`` lowers to ``mov ax, STR_ORG`` (NASM resolves the label address at assembly time from the trailing inline-asm block's ``db`` definitions).  ``tests/bboeos.h`` gains the matching ``extern char STR_*[];`` declarations so clang's syntax check accepts the char-pointer arg.  The new ``regparm(1) + carry_return match_word_c(char *keyword)`` wrapper is a 2-instruction naked-asm body (``mov di, ax ; call match_word``) parallel to the three specialized ``match_word_c_str_*`` wrappers; ``parse_directive`` uses the generic form for all eight keyword probes instead of eight more specialized wrappers.  ``dw`` / ``dd`` emit their 2/4 bytes via paired ``emit_byte`` calls (low then high, with two trailing zeros for ``dd``) instead of the retired ``emit_word_ax`` naked-register helper.  ``times N db`` saves the post-``db`` source cursor to a local and resets it before each ``parse_db()`` iteration so the same operands are repeated N times.  Binary grows 145 bytes (10983 → 11128): bp frame + the if-chain's accumulated ``push ax`` / ``pop ax`` around each ``match_word_c`` call (the specialized wrappers avoided this since cc.py's naked-asm elide doesn't apply to ``regparm(1)`` parametric calls).  1 inline-asm block retired

- asm.c: port ``parse_line`` to pure C.  Top-level line dispatcher: ``source_cursor = line_buffer; skip_ws();`` at entry, early-return on empty / ``;`` comment / ``%`` directive, else scan for ``:`` (colon-terminated label), ``' '``/``'\t'`` (potential ``NAME equ VALUE``), or ``'\0'`` (not a label).
