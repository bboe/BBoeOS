# Changelog

All notable changes to BBoeOS are documented in this file. Dates reflect
when changes landed, grouped under the version that was (or will be) current
at the time.

## [Unreleased](https://github.com/bboe/BBoeOS/compare/5156ae9...main)

- asm.c: port ``parse_line`` to pure C.  Top-level line dispatcher: ``source_cursor = line_buffer; skip_ws();`` at entry, early-return on empty / ``;`` comment / ``%`` directive, else scan for ``:`` (colon-terminated label), ``' '``/``'\t'`` (potential ``NAME equ VALUE``), or ``'\0'`` (not a label).  The colon path null-terminates at ``:``, dispatches on ``pass == 1`` (symbol_set with ``is_local`` picking ``global_scope`` vs ``0xFFFF``) or ``pass == 2`` global-label refresh (symbol_lookup ŌåÆ update ``global_scope``), restores the colon, and falls through to ``parse_directive`` for any trailing content.  The equ path uses a new ``match_word_c_str_equ`` wrapper (parallel to ``_str_word`` / ``_str_short``); on match, null-terminates the name at the first whitespace, resolves the value, and on pass 1 calls a new ``regparm(1) symbol_add_constant_c(value)`` naked-asm wrapper that sets ``BX = 0xFFFF`` before invoking ``symbol_add_constant`` (caller pre-loads ``SI = label_start``).  On miss, source_cursor rewinds to ``label_start`` and parse_directive gets its crack at the token.  cc.py: ``generate_index_assign`` gains the SI-scratch guard that ``generate_expression``'s subscript-read path already had ŌĆö without it, ``space_pos[0] = '\0';`` lowered to ``mov si, [bp-N] ; mov byte [si], 0`` clobbered the SI alias of ``source_cursor``, leaving the post-equ source cursor reset to the pre-null whitespace and resolve_value reading garbage zero.  Guard goes *outside* the ``push ax`` / ``pop ax`` save pair so the pop order matches the push order (push ax ŌåÆ push si ŌåÆ pop ax ŌåÆ pop si).  ``peephole_index_through_memory`` can't fuse through the added ``push si`` / ``pop si`` wrapping, so programs with an ``asm_register("si")`` global pay ~2 bytes per non-pinned subscript assignment (other programs see zero impact, the guard returns False when no SI alias exists).  Binary grows 104 bytes (10879 ŌåÆ 10983): bp frame + 6 locals (label_start, c, colon_pos, is_local, space_pos, value) + the ``||`` fan-out for the two early-return checks + the colon-path pass / local-ness dispatch + the new symbol_add_constant_c wrapper + the cc.py-wide SI-guard overhead on ``source_buffer[source_buffer_position]`` / ``line_buffer[length]`` / ``output_buffer[output_position]`` subscript-assigns (each gains a paired push/pop the peephole can no longer fuse).  Self-host test still byte-identical on all 31 static programs

- asm.c: port ``parse_db`` to pure C.  The three operand shapes ŌĆö a backtick string (``\`...\```) with ``\n`` / ``\0`` / ``\t`` / ``\r`` / ``\e`` / ``\\`` / ``\x??`` escapes and pass-through for any other ``\c``; a multi-byte single-quoted string (``'foo'``); and the default value path (numbers, symbols, single-char ``'c'`` literals that fall through to resolve_value) ŌĆö collapse into a ``while (1)`` outer loop with an ``if / else if / else`` operand dispatch.  The ``cmp byte [si+2], 27h`` single-char-vs-string disambiguator becomes a straight ``source_cursor[2] != '\''`` (subscript on the SI-pinned global needs no scratch-register guard).  Each byte goes through ``emit_byte(value)`` directly ŌĆö the retired asm's ``call emit_byte_al`` sites already route through the same pure-C function via the AL-in thunk, so the behavior matches.  cc.py's ``CHARACTER_ESCAPES`` table gains ``\'`` (0x27) and ``\e`` (0x1B) so the port can spell ``'\''`` / ``'\e'`` naturally instead of falling back to numeric literals that would fail ``validate_comparison_types``' char-vs-int guard.  ``hex_digit`` and ``resolve_value`` are reached through their existing pure-C signatures (regparm(1) int for the former, plain int for the latter).  Binary grows 62 bytes (10817 ŌåÆ 10879): bp frame + the larger if/else fan-out over the escape-code dispatch vs. the retired asm's shared-tail-``.hdb_esc_emit`` idiom, partially offset by the eliminated push/pop save of AX around the unknown-escape fallback (the pure-C version spills ``esc`` to a local that survives the intervening ``emit_byte('\\')`` call).  1 inline-asm block retired

- cc.py: booleanize comparison BinOps used as expression values (``int x = (a == b);`` / ``foo((c > 0))`` / ``(a != c) + (b != c)``).  Prior codegen emitted ``cmp ax, cx ; mov ax, 0`` and dropped the compare result, landing 0 regardless of inputs.  Fixed by extending the BinOp-comparison arm to ``cmp ax, cx ; mov ax, 0 ; <jump_when_false> .bool_N ; inc ax ; .bool_N:`` ŌĆö 6 extra bytes over the buggy output, and ``mov ax, 0`` (not ``xor ax, ax``) so the ``cmp``'s flags survive the zero-fill.  Conditions in ``if`` / ``while`` still go through ``emit_condition`` and skip the booleanization entirely (no ``cmp`` is emitted twice).  New ``src/c/booltest.c`` smoke test exercises all six comparison operators plus an arithmetic combination (``(a == b) + (c > a) + (a != c) == 3``); added to the test_programs runtime suite and the test_asm self-host suite (via the generic ``compile_c_sources`` path).  ``peek_label_target``'s ``is_local`` collapses from the split ``int is_local = 0; if (source_cursor[0] == '.') is_local = 1;`` back to the one-liner ``int is_local = (source_cursor[0] == '.');`` ŌĆö byte-identical output, one fewer statement

- asm.c: port ``peek_label_target`` to pure C via ``carry_return`` + a new ``peek_label_value`` global carrying the AX-side of the retired dual AX + CF return.  ``carry_return`` can only signal CF, so the two ``encode_rel8_jump`` call sites (still inline asm) gain a ``mov ax, [_g_peek_label_value]`` after the ``jnc`` to recover the target address.  Pure-C body walks ``source_cursor`` (SI pin) forward through the identifier, null-terminates in place, resets ``source_cursor`` to the name start for ``symbol_lookup``'s SI = name ABI, then restores the saved delimiter via a local ``end_pos`` char-pointer (cc.py's SI scratch guard preserves the cursor across the subscript).  ``is_local`` is captured as ``if (source_cursor[0] == '.') is_local = 1;`` ŌĆö cc.py's codegen drops the comparison on the shorter ``int is_local = (expr == val);`` form (booleanization-from-expression isn't yet lowered), turning every program's short-form jumps into near-form encodings; the split ``if``/``= 1`` pattern matches ``handle_unknown_word``'s precedent.  The char-class chain lives in an ``if`` condition (not an int-assignment RHS) to stay within cc.py's supported expression shapes.  The symbol_lookup call with its SI = name / BX = scope ABI stays as a two-line inline-asm block inside each branch of the ``is_local`` dispatch.  Binary grows 74 bytes (10741 ŌåÆ 10815): bp frame + 5 locals (spilled saved, end_pos, delim, c, is_local) + the is_local split + the char-class ``||`` chain fan-out + cc.py's fastcall call-site shim for the two inline asm blocks, minus the 2 ├Ś 4-byte ``mov ax, [_g_peek_label_value]`` added at each caller site.  1 inline-asm block retired

- asm.c: port ``handle_jmp`` to pure C ŌĆö the last ``handle_*`` with an inline-asm body.  The retired asm's ``.hj_no_short`` branch (``pop si ; push si ; call skip_ws`` on match failure) was a no-op since ``match_word`` rewinds SI on miss and ``skip_ws`` is idempotent; the pure-C port collapses both paths into ``skip_ws() ; match_word_c_str_short() ; encode_rel8_jump(0xEB)``.  New ``match_word_c_str_short`` ``carry_return`` wrapper hard-codes ``mov di, STR_SHORT ; call match_word``, parallel to the existing ``match_word_c_str_word`` ŌĆö the caller discards the CF because either outcome falls through to the same ``encode_rel8_jump`` tail.  cc.py handles the bare ``Call`` statement by generating the call with ``discard_return=True``.  Binary shrinks 6 bytes (10747 ŌåÆ 10741): the pure-C handler's bp frame plus two calls outweighs the retired asm's 7-instruction body by less than the redundant ``.hj_no_short`` path's savings.  1 inline-asm block retired; ``handle_*`` family now 100% pure C (for real this time ŌĆö the previous milestone missed ``handle_jmp``)

- asm.c: port ``mem_op_reg_emit`` to pure C via ``regparm(1)``.  Shared helper for the ``<op> [disp16], r16`` memory-destination form (3 callers: handle_add / handle_and / handle_sub).  Opcode arrives via AX; body walks ``source_cursor`` past ``[`` / ``disp`` / ``]`` / ``,`` / ``<reg>`` using the existing ``resolve_value`` and ``parse_register_optional`` helpers, then emits ``<opcode> <modrm(mod=00, reg=src, rm=110)> <disp16>``.  Malformed structure calls ``abort_unknown()`` (which never returns).  The three callers drop their ``asm("mov al, OP ; call mem_op_reg_emit")`` shim in favor of ``mem_op_reg_emit(OP)``.  Binary grows 42 bytes (10705 ŌåÆ 10747): the bp frame + local-for-disp spill + register-convention AX-through-parameter fixup outweigh the shim savings at each call site.  4 inline-asm blocks retired (1 body + 3 shims)

- asm.c: port ``handle_unknown_word`` to pure C ŌĆö the last remaining handler with a file-scope inline-asm body.  Scans ``source_cursor`` past the bare-label name, null-terminates in place, and rewinds ``source_cursor`` to the name start so the two symbol-table entry points (``symbol_set`` on pass 1, ``symbol_lookup`` on pass 2) still see SI pointing at the name ŌĆö both calls live in tight inline-asm blocks that set AX / BX and invoke the inline-asm entry.  Pass 1 adds the symbol with the current address as the value (scope = 0xFFFF for global, ``global_scope`` for ``.``-prefixed locals) and updates ``global_scope`` to the new entry's index; pass 2 refreshes ``global_scope`` via ``symbol_lookup`` for globals (locals skip).  After the symbol-table update, the source cursor jumps past the null to whatever trailing text was on the line and ``parse_directive`` gets another crack at it.  Character-class compares use ``'\t'`` / ``'\0'`` char literals instead of numeric ``9`` / ``0`` to dodge cc.py's int-vs-char validation.  Binary grows 53 bytes (10652 ŌåÆ 10705) ŌĆö modest given the handler's branching: the pass / is_local dispatch + two tiny inline-asm blocks for ``symbol_set`` / ``symbol_lookup``.  1 inline-asm block retired; ``handle_*`` family now 100% pure C

- asm.c: port ``handle_mov`` to pure C.  The ten operand shapes ŌĆö ``mov es, r16`` segment write (8E /r), reg-reg (88/89 modrm), reg-imm (B0+r / B8+r short), reg-[disp16] (A0/A1 short for AL/AX else 8A/8B + modrm rm=110), reg-[reg+disp] (8A/8B + modrm mod=00/01/10), [disp16]-reg (A2/A3 short for AL/AX else 88/89 + modrm), [disp16]-imm (C6/C7 + modrm rm=110 + disp16 + imm), [reg+disp]-reg (88/89 + modrm + disp), [reg+disp]-imm (C6/C7 + modrm + disp + imm) ŌĆö collapse into a nested ``if t1 then (switch t2)`` dispatch.  ``op1_size`` is captured *after* both ``parse_operand_c`` calls, matching the retired asm's side-effect semantics: when op1 is memory without a ``byte`` / ``word`` prefix and op2 is a register, the register's size set by op2's parse_operand carries forward as the mov's width.  Snapshotting op1_size *before* op2's parse_operand (my first attempt) produced stale 8-bit encodings ŌĆö uptime.asm's ``mov [counter], ax`` dropped from the A3 short form to the A2 short form across the whole self-host run.  The ``mov es, r16`` peek-then-fall-through uses the standard ``char *saved = source_cursor`` + restore idiom when the second byte isn't ``,``.  Binary grows 173 bytes (10479 ŌåÆ 10652): the expanded dispatch tree plus cc.py's packed-return decode locals plus the duplicated ``reg_to_rm`` + disp-size-switch blocks for the [reg+disp] emits.  1 inline-asm block retired

- asm.c: port ``handle_call`` / ``handle_movzx`` / ``handle_sbb`` to pure C.  ``handle_movzx`` uses ``parse_register_c`` for the destination register and ``parse_operand_c`` for the source; the two branches cover register source (``0F B6 modrm(reg=dst, rm=src)``) and indirect memory source (``0F B6 modrm(reg=dst, rm=reg_to_rm(src_reg), mod=00/01)``).  ``handle_call`` handles the direct ``E8 rel16`` form (``target - current_address - 2``) and the indirect ``FF /2 [reg+disp8]`` form; malformed indirect (bad type / zero disp / disp out of ``-128..127``) calls ``abort_unknown()``.  ``handle_sbb`` keeps the single ``sbb word [disp16], imm8`` form the TCP-checksum fold needs ŌĆö uses the new ``match_word_c_str_word`` wrapper to peel the ``word`` prefix, then resolves the disp/imm and emits ``83 /3 [disp16] imm8``.  ``resolve_label`` gets a forward declaration for clang's declare-before-use rule.  Binary grows 136 bytes (10343 ŌåÆ 10479) ŌĆö ~45 bytes per handler, lighter than the full ``parse_operand_c`` dispatch-five-ways family.  3 inline-asm blocks retired

- asm.c: port ``handle_add`` and ``handle_sub`` to pure C.  Both share the now-standard reg-reg / reg-[disp16] / reg-[reg+disp] / reg-imm dispatch on the ``parse_operand_c`` packed return (type 0 / 2 / 3 / else); ``handle_add`` adds the AX-short 05 imm16 and AL-short 04 imm8 forms (``add`` is the only family member with both short forms).  Both handlers' ``[disp16], r16`` memory-destination branch keeps a 2-line inline-asm shim forwarding to ``mem_op_reg_emit`` (still inline).  ``handle_sub`` also has the ``word [disp16], imm16`` path (81 /5 iw for TCP-checksum folds) that relies on ``match_word`` to peel the ``word`` keyword off SI; a new ``match_word_c_str_word`` ``carry_return`` wrapper hard-codes ``mov di, STR_WORD ; call match_word`` so the caller can dispatch via ``if (match_word_c_str_word()) { ... }``.  On mismatch the caller falls through to ``abort_unknown()`` when structure is bad; cc.py emits the ``call abort_unknown ; ret`` sequence whose ``ret`` is dead code since abort_unknown's tail ``jmp`` never returns.  Binary grows 298 bytes (10045 ŌåÆ 10343): two bp frames plus the packed-return decode locals plus the five-way branch fan.  2 inline-asm blocks retired

- asm.c: port ``handle_and`` / ``handle_cmp`` / ``handle_or`` / ``handle_test`` / ``handle_xor`` to pure C on top of the existing ``parse_operand_c`` / ``parse_register_c`` / ``parse_register_optional`` helpers.  All five share the reg-reg / reg-imm / r-mem dispatch shape and differ only in opcode pairs and /r constants ŌĆö ``and`` 20/21/24/80/83 /4, ``or`` 08/09/0C/80/83 /1, ``xor`` 30/31/34/80/83 /6, ``cmp`` 38/39/3C/3D/80/81/83 /7, ``test`` 84/85/A8/A9/F6/F7 /0.  ``handle_and``'s ``[disp16], r16`` memory-destination branch stays a two-line inline-asm ``mov al, 0x21 ; call mem_op_reg_emit`` ŌĆö porting ``mem_op_reg_emit`` itself is a follow-up.  ``handle_cmp``'s edge cases collapse: its saved ``cmp_op1_size`` / ``cmp_imm_byte`` globals (stashed across the intervening ``parse_register`` call because the retired asm's shared tail-``call parse_operand`` would clobber ``op1_size``) become pure-C locals that cc.py spills to the bp frame naturally.  The two obsolete globals and their ``_g_`` aliases in the trailing %define block retire.  Signed-range ``value >= -128 && value <= 127`` checks replace the asm's ``add dx, 0x80 ; cmp dx, 0xFF ; ja`` idiom (same bug class as ``handle_push``).  Binary grows 586 bytes (9459 ŌåÆ 10045): each handler picks up a bp frame plus cc.py's packed-return decode locals plus the three-way-type branch fan vs. the retired asm's local-label shared-tail idiom.  5 inline-asm blocks retired

- asm.c: port ``handle_inc`` and ``handle_dec`` to pure C via a new ``parse_operand_c`` wrapper.  The retired ``parse_operand`` returned three values across AX and DX (AH = type, AL = reg, DX = displacement / immediate), which doesn't fit cc.py's AX-only return ABI; the wrapper stashes DX into a new ``parse_operand_value`` global before falling through, so callers write ``int po = parse_operand_c();`` and read the three fields via ``(po >> 8) & 0xFF`` / ``po & 0xFF`` / ``parse_operand_value``.  ``op1_size`` keeps being set by ``parse_operand`` as a side effect ŌĆö C callers read it through the existing global.  ``handle_inc`` / ``handle_dec`` branch identically on ``type`` (0 = register / 2 = disp16 / 3 = reg+disp) and ``op1_size`` (8 vs 16) and differ only in the /r-field constants (inc: 0x40+reg for r16, 0xC0|reg for r8, 0x06 for disp16 modrm, bare rm for indirect; dec: 0x48+reg, 0xC8|reg, 0x0E, rm|0x08).  A ``regparm(1) int reg_to_rm(int reg)`` forward declaration satisfies clang's declare-before-use rule for both handlers.  Binary grows 165 bytes (9294 ŌåÆ 9459): two bp frames plus the packed-return / side-effect-global decode locals plus the common three-way branch fan vs the retired asm's shared-code-via-fall-through form.  2 inline-asm blocks retired

- asm.c: port ``handle_push`` to pure C via a new ``parse_register_optional`` wrapper.  The carry_return ``parse_register_c`` wrapper discards AX on its way through the ``if`` dispatch, so a handler that needs both the success signal and the parsed register can't use it; the new ``parse_register_optional`` returns ``-1`` on failure (safe sentinel since success packs ``reg 0..7`` + ``size 8 or 16`` into the ``0x0800 .. 0x1007`` range) and the parsed AX otherwise.  handle_push saves the source cursor, tries ``parse_register_optional``, emits ``50+reg`` on success or restores the cursor and calls ``resolve_value`` for the imm16 path.  The short ``6A imm8`` form uses a signed-fits-imm8 check (``value >= -128 && value <= 127``) to match the retired asm's ``add dx, 0x80 ; cmp dx, 0xFF ; ja`` ŌĆö the arithmetic-overflow + unsigned-compare idiom can't translate to cc.py's signed ``jg``, which falsely classifies values like ``SECTOR_BUFFER`` (0xE000 ŌåÆ -8192 signed) as imm8-fitting; the ``value >= -128 && value <= 127`` pair compiles to a paired signed compare that agrees with the unsigned range.  Segment-register ``push ds`` / ``push es`` use the same ``cmp word [si], 0x7364``/``0x7365`` fusion cc.py already applied to ``handle_pop``.  Binary grows 55 bytes (9239 ŌåÆ 9294): bp frame + wrapper overhead + imm8-range locals

- asm.c: port ``handle_xchg`` and ``handle_pop`` to pure C.  ``handle_xchg`` uses ``parse_register_c`` for both operands and the existing ``make_modrm_reg_reg_impl(reg, rm)`` 2-arg fastcall helper for the long-form encoding; a forward declaration of ``make_modrm_reg_reg_impl`` near the top of asm.c satisfies clang's declare-before-use rule.  ``handle_pop`` peeks ``source_cursor[0..1]`` for the ``ds`` / ``es`` segment-register shortcuts (1F/07) ŌĆö cc.py fuses the two adjacent byte compares into a single ``cmp word [si], 0x7364`` ŌĆö then falls through to ``parse_register_c()`` and ``emit_byte(0x58 | (pr & 0xFF))`` for the general register form.  Binary grows 103 bytes (9136 ŌåÆ 9239): each handler gains a bp frame plus cc.py's stack-spill of the packed-register locals (``pr1`` / ``reg1`` / ``size1`` / ``pr2`` / ``reg2`` for xchg; ``pr`` for pop) vs the retired inline-asm BX-in-register shape.  2 inline-asm blocks retired

- asm.c: port 7 handlers (``handle_mul`` / ``handle_neg`` / ``handle_not`` / ``handle_div`` / ``handle_adc`` / ``handle_shl`` / ``handle_shr``) to pure C via a new ``parse_register_c`` wrapper.  The wrapper is a naked-asm function with ``carry_return`` + ``asm("call parse_register")`` body ŌĆö AX (packed ``AH = size``, ``AL = reg number``) propagates through cc.py's naked ``ret``, so pure-C callers write ``int pr = parse_register_c();`` and extract via ``pr & 0xff`` / ``pr >> 8``.  The wrapper sidesteps the "carry_return can't also return AX" tension by omitting explicit ``return 0 / 1`` ŌĆö the inner call's AX and CF both reach the caller through the naked ``ret``.  ``handle_mul`` / ``neg`` / ``not`` / ``div`` share the same shape (``emit F6/F7 byte based on AH ; emit modrm = 0xN0 | reg``); ``handle_adc`` is the single ``adc r16, imm8`` checksum-fold idiom; ``handle_shl`` / ``shr`` handle both the short D0/D1 form (count == 1) and the imm8 C0/C1 form.  7 inline-asm blocks retired; binary grows 273 bytes (8863 ŌåÆ 9136) ŌĆö ~39 bytes per handler for the bp frame, packed-register local, and multi-branch opcode selection

- asm.c: port ``parse_number`` to pure C.  Three-branch body (hex prefix ``0x...``, hex suffix ``...h``, decimal) becomes three ``while (1) { ... return }`` loops after a peek-ahead that detects the hex-suffix case by scanning the digit run to its terminator.  The pure-C body leans on the now-pure-C ``hex_digit`` (returning ``-1`` for non-hex) so each digit loop is ``d = hex_digit(source_cursor[0]); if (d < 0) return value; value = (value << 4) | d;``.  No register-preservation thunk ŌĆö ``resolve_value`` (sole caller) already saves BX/CX/DI at its own entry, so parse_number clobbering them is within scope.  ASCII class comparisons use numeric literals (``c >= 48 && c <= 57``) to dodge cc.py's int-vs-Char type check.  Binary grows 107 bytes (8756 ŌåÆ 8863): the pure-C body's bp frame + two locals + multiple early-return epilogue sequences + char-through-int comparison load pattern for each character check.  1 inline-asm block retired

- asm.c: port ``hex_digit`` to pure C via ``regparm(1)``.  New ABI: pass char in AX (caller zero-extends with ``xor ah, ah``), return numeric value in AX (0..15) or ``-1`` for non-hex; callers check ``js`` after a ``test ax, ax`` instead of the retired ``jc`` on CF.  Pure-C body is a three-range sign-extended compare chain (digit / upper / lower).  4 call sites update: 2 in ``parse_db``'s ``\x..`` escape handler (straight translation, 1 extra ``xor ah, ah`` per site, 1 fewer instruction on the store path) and 2 in ``parse_number``'s hex-prefix / hex-suffix loops (accumulator moves from AX to BX so the AX-passing hex_digit handshake doesn't clobber it ŌĆö BX is scratch inside parse_number since its entry spills BX/CX/DX to the stack).  Hex literals in the compares expand to numeric decimals (``>= 48 && <= 57`` etc.) because cc.py's ``validate_comparison_types`` rejects ``int`` vs ``Char`` mismatch on ``c >= '0'``.  Binary grows 43 bytes (8713 ŌåÆ 8756): pure-C body's bp frame + chained cmp/branch pairs (~40 bytes) + 4 caller-site updates (~3 bytes each, net) vs. the retired ~20-byte naked-asm body.  1 inline-asm block retired

- asm.c: port ``make_modrm_reg_reg`` to pure C via ``regparm(1)`` with a 2-arg fastcall signature.  Pure-C ``make_modrm_reg_reg_impl(reg, rm)`` receives ``reg`` in AX and ``rm`` on the stack; returns ``0xC0 | (reg << 3) | rm`` in AX.  The legacy ``make_modrm_reg_reg`` label becomes a naked-asm thunk that pushes BX (which carries BL = ``rm`` in the caller's ABI), calls the impl, and cleans up the pushed word ŌĆö preserves the AL-in / BL-in / AL-out ABI for the 7 inline-asm caller sites unchanged.  First 2-arg fastcall port, validating cc.py's regparm(1)-with-stack-args handshake under the real self-host workload.  Binary grows 42 bytes (8671 ŌåÆ 8713) ŌĆö the 8-byte thunk plus the ~34-byte pure-C impl (bp frame + byte-mask pair + push-pop-or compute + epilogue) vs. the retired 7-byte naked-asm body.  1 inline-asm block retired

### [2026-04-19](https://github.com/bboe/BBoeOS/compare/bf10828...main)

- asm.c: port ``resolve_label`` to pure C.  The two-branch body ŌĆö pass 2 falls through to ``resolve_value()``, pass 1 scans ``source_cursor`` past identifier characters and returns ``current_address`` as a placeholder ŌĆö becomes a C ``if (pass != 1) return resolve_value();`` then a ``while`` loop over the source cursor.  ``source_cursor``'s SI alias lets the loop read ``mov al, [si]`` directly, and ``source_cursor = source_cursor + 1`` folds to ``inc si`` via the existing register-arithmetic peephole.  The 3 inline-asm callers (two in ``encode_rel8_jump`` short / long paths, one in ``handle_call``) already read AX after the call, so no caller-side changes are needed.  Binary grows 28 bytes (8643 ŌåÆ 8671): cc.py's ``&&`` / ``||`` chain over the five character-class checks inflates the scan loop, but the result is still self-host byte-identical.  1 inline-asm block retired
- asm.c: port ``symbol_entry_address`` to pure C via ``regparm(1)``.  The one-line body (``return index * SYMBOL_ENTRY``) replaces the inline-asm ``mov bx, SYMBOL_ENTRY ; mul bx ; mov di, ax`` sequence; cc.py lowers it through ``mul bx`` (same instruction) but now adds a bp frame plus the full ``mov sp, bp / pop bp / ret`` epilogue.  Return moves from DI to AX, so the 4 inline-asm callers each gain a ``mov di, ax`` after the call ŌĆö 2 bytes ├Ś 4 sites = 8 bytes ŌĆö to preserve the DI-out ABI the callers' ES:DI symbol-table indexing expects.  Binary grows 16 bytes (8627 ŌåÆ 8643).  1 inline-asm block retired
- asm.c: port ``reg_to_rm`` to pure C via the ``regparm(1)`` fastcall convention.  The four-way switch (bx ŌåÆ 7, si ŌåÆ 4, di ŌåÆ 5, default ŌåÆ 6) lowers to a chain of ``cmp word [bp-2], N ; jne`` branches in pure C.  Inline-asm callers continue to do ``mov al, X ; call reg_to_rm`` unchanged ŌĆö AX arrives with AH carrying whatever junk the caller left there, so the body masks with ``reg = reg & 0xFF`` before the switch to match the old AL-only comparison semantics.  The pure-C version tests byte-identical on every ``static/`` program through the self-hosted assembler (a real end-to-end smoke test of the ``regparm(1)`` + fastcall + scratch-register-guard stack under the real assembler's workload).
